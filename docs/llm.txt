# Agent-Native Middleware API — LLM-Optimized Documentation
# Version: 0.1.0
# Last Updated: 2026-02-22
# Purpose: Enable AI agents to discover, authenticate, and consume this API.

## WHAT THIS API DOES (ONE SENTENCE)
Headless middleware that gives AI agents secure IoT device control, self-healing code via telemetry analysis, automated viral video distribution, and agent-to-agent messaging — all via REST API with zero GUI.

## BASE URL
- Production: https://api.yourdomain.com
- Local: http://localhost:8000
- OpenAPI Spec: GET /openapi.json
- Interactive Docs: GET /docs

## AUTHENTICATION
- Method: API Key
- Header: X-API-Key
- Example: `curl -H "X-API-Key: your-key-here" https://api.yourdomain.com/v1/iot/devices`
- Error (401): Missing key → `{"error": "missing_api_key"}`
- Error (403): Invalid key → `{"error": "invalid_api_key"}`

## SERVICE 1: IoT PROTOCOL BRIDGE (/v1/iot)

### Purpose
Translate legacy IoT protocols (MQTT, CoAP, Zigbee, Z-Wave, Modbus, BLE) into a unified REST API with mandatory topic-level access controls.

### Why topic ACLs matter
Without ACLs, an agent connecting to an MQTT broker could accidentally access camera feeds, microphones, and floor plans for thousands of devices. This API enforces deny-by-default.

### Endpoints
- POST /v1/iot/devices → Register a device with protocol type and ACL rules
- GET /v1/iot/devices → List all registered devices (paginated)
- GET /v1/iot/devices/{device_id} → Get device details and bridge endpoint
- DELETE /v1/iot/devices/{device_id} → Deregister a device
- POST /v1/iot/devices/{device_id}/messages → Send a message through the bridge
- POST /v1/iot/devices/{device_id}/subscribe → Subscribe to device messages

### Example: Register a device with ACLs
```json
POST /v1/iot/devices
{
  "device_id": "warehouse-sensor-042",
  "protocol": "mqtt",
  "topic_acl": {
    "device/+/telemetry": "read",
    "device/+/command": "write",
    "device/+/camera": "deny"
  }
}
```

### Example: Send a command
```json
POST /v1/iot/devices/warehouse-sensor-042/messages
{
  "topic": "device/warehouse-sensor-042/command",
  "payload": {"action": "report_status"},
  "qos": 1
}
```

## SERVICE 2: AUTONOMOUS PRODUCT MANAGER (/v1/telemetry)

### Purpose
Ingest raw telemetry (errors, sessions, LLM traces, API logs), detect anomalies, and autonomously generate pull requests to fix bugs.

### How it works
1. Agent sends telemetry events via batch or single endpoint
2. System analyzes patterns to detect error spikes, latency regressions, missing features
3. Agent queries anomalies and optionally triggers auto-PR generation
4. LLM generates targeted code fix, runs tests, pushes PR

### Endpoints
- POST /v1/telemetry/events → Batch ingest (up to 1000 events)
- POST /v1/telemetry/events/single → Single event convenience endpoint
- GET /v1/telemetry/anomalies → List detected anomalies (filterable by severity)
- GET /v1/telemetry/anomalies/{anomaly_id} → Anomaly details with suggested fix
- POST /v1/telemetry/anomalies/{anomaly_id}/auto-pr → Generate autonomous PR
- GET /v1/telemetry/stats → Aggregate telemetry statistics

### Event Types
- error: Application errors with stack traces
- warning: Non-fatal issues
- session: User/agent session data
- api_call: API request/response logs
- llm_trace: LLM interaction traces (prompts, completions, latency)
- performance: Latency, throughput, resource usage
- custom: Arbitrary structured events

### Example: Batch ingest
```json
POST /v1/telemetry/events
{
  "events": [
    {
      "event_type": "error",
      "source": "iot-bridge",
      "message": "MQTT connection timeout after 30s",
      "severity": "high",
      "stack_trace": "Traceback (most recent call last):\n  ...",
      "metadata": {"device_id": "sensor-042", "broker": "mqtt://192.168.1.50"}
    }
  ]
}
```

### Example: Generate auto-PR (dry run)
```json
POST /v1/telemetry/anomalies/anom-123/auto-pr
{
  "anomaly_id": "anom-123",
  "dry_run": true
}
```
Returns proposed diff without pushing.

## SERVICE 3: PROGRAMMATIC MEDIA ENGINE (/v1/media)

### Purpose
Ingest long-form video, detect viral hooks, reframe for vertical formats, generate animated captions, and distribute to social platforms — entirely via API.

### Pipeline
1. Upload video (URL or direct upload)
2. Engine detects viral hooks (emotional peaks, speech patterns, visual surprises)
3. Generate platform-ready clips (reframed, captioned)
4. Distribute to YouTube Shorts, TikTok, Instagram Reels, X, LinkedIn

### Why captions are mandatory
80% of social video is consumed with sound off. Animated captions are not optional — they are required for engagement.

### Endpoints
- POST /v1/media/videos → Upload video for processing
- GET /v1/media/videos/{video_id} → Check processing status
- GET /v1/media/videos/{video_id}/hooks → Get detected viral hooks
- POST /v1/media/videos/{video_id}/clips → Generate reframed, captioned clips
- POST /v1/media/distribute → Push clips to social platforms
- GET /v1/media/clips/{clip_id} → Get clip details and download URL

### Supported platforms
- youtube_shorts
- tiktok
- instagram_reels
- x_video
- linkedin_video

### Caption styles
- word_by_word: Each word animates as spoken
- sentence_highlight: Full sentence with active word highlighted
- karaoke: Karaoke-style progressive reveal
- minimal_subtitle: Simple bottom subtitle

### Example: Full pipeline
```json
# 1. Upload
POST /v1/media/videos
{"source_url": "https://storage.example.com/demo.mp4", "title": "Q1 Demo"}

# 2. Get hooks (after processing)
GET /v1/media/videos/{video_id}/hooks?min_confidence=0.7

# 3. Generate clips
POST /v1/media/videos/{video_id}/clips
{"max_clips": 5, "aspect_ratios": ["9:16"], "caption_style": "word_by_word"}

# 4. Distribute
POST /v1/media/distribute
{"clip_ids": ["clip-1", "clip-2"], "platforms": ["youtube_shorts", "tiktok"], "title": "Check this out", "optimize_schedule": true}
```

## SERVICE 4: AGENT COMMUNICATIONS (/v1/comms)

### Purpose
Machine-to-machine messaging that bypasses human-first platforms (Gmail, Slack). Agents register, discover each other by capability, send structured messages, and hand off tasks to specialists.

### Why not Gmail/Slack?
- Human spam filters block automated senders
- Rate limits designed for humans (~500/day) throttle agent workloads
- OAuth flows require browser interaction (violates Zero-GUI rule)

### Endpoints
- POST /v1/comms/agents → Register an agent with name, capabilities, optional webhook
- GET /v1/comms/agents → List agents, optionally filtered by capability
- POST /v1/comms/messages → Send a structured message to another agent
- GET /v1/comms/messages/{agent_id}/inbox → Poll for pending messages
- POST /v1/comms/messages/{agent_id}/ack/{message_id} → Acknowledge a message
- POST /v1/comms/handoff → Hand off a task to an agent with a specific capability

### Message Types
- request: Asking another agent to do something
- response: Reply to a request
- event: Notification of something that happened
- heartbeat: Liveness check
- handoff: Transfer responsibility to another agent

### Priority Levels
- critical: Immediate delivery, aggressive retry
- high: Deliver within 1 minute, 2x retry
- normal: Deliver within 5 minutes, 1x retry
- low: Batched delivery, best effort

### Example: Register an agent
```json
POST /v1/comms/agents
{
  "name": "iot-monitor-agent",
  "capabilities": ["iot-monitoring", "mqtt-bridging", "device-management"],
  "webhook_url": "https://my-agent.example.com/webhook/messages"
}
```

### Example: Capability-based handoff (swarm intelligence)
```json
POST /v1/comms/handoff?from_agent=orchestrator-001
{
  "capability": "video-transcription",
  "context": {"video_id": "abc-123", "language": "en"}
}
```
System finds the best available agent with that capability and routes the task.

## SERVICE 5: CONTENT FACTORY & ALGORITHMIC SCHEDULING (/v1/factory)

### Purpose
Turn one source asset into dozens of format-adapted content pieces, then use engagement analytics to algorithmically schedule distribution for maximum reach.

### How it works
1. Submit a source (video clip ID or URL) with desired output formats
2. Factory generates format-adapted pieces (short videos, images, text posts, audiograms, carousels, blog excerpts, email snippets)
3. Feed engagement data from platform APIs into the scheduler
4. Scheduler recommends optimal posting times per platform

### Output Formats
- short_video: 15-60s vertical (Shorts/Reels/TikTok) — 5 pieces per source
- long_video: 2-15min horizontal (YouTube) — 1 piece
- audiogram: Audio waveform video for podcasts — 1 piece
- carousel: Multi-image slideshow (Instagram/LinkedIn) — 2 pieces
- static_image: Single thumbnail/poster with text overlay — 5 pieces
- text_post: Platform-native text (X/LinkedIn) — 5 pieces
- blog_excerpt: SEO-optimized text snippet — 1 piece
- email_snippet: Newsletter-ready HTML block — 1 piece

### Endpoints
- POST /v1/factory/pipelines → Create content generation pipeline (async, returns 202)
- GET /v1/factory/pipelines/{pipeline_id} → Check pipeline status
- GET /v1/factory/pipelines/{pipeline_id}/content → List generated content pieces
- GET /v1/factory/content/{content_id} → Get content piece details and download URL
- POST /v1/factory/analytics → Ingest engagement metrics (up to 500 per batch)
- GET /v1/factory/analytics/summary → View analytics summary
- POST /v1/factory/schedule → Get algorithmically-optimized posting schedule

### Example: Full content pipeline
```json
# 1. Create pipeline
POST /v1/factory/pipelines
{
  "title": "Q1 Product Launch",
  "source_url": "https://storage.example.com/launch.mp4",
  "target_formats": ["short_video", "static_image", "text_post", "email_snippet"],
  "brand_config": {"primary_color": "#FF6B00", "font_family": "Inter"},
  "auto_schedule": true
}

# 2. Check status (after processing)
GET /v1/factory/pipelines/{pipeline_id}

# 3. List generated pieces
GET /v1/factory/pipelines/{pipeline_id}/content

# 4. Feed engagement data
POST /v1/factory/analytics
{
  "metrics": [
    {"platform": "tiktok", "metric_type": "views", "value": 15000, "recorded_at": "2026-02-20T14:00:00Z"}
  ]
}

# 5. Get optimal schedule
POST /v1/factory/schedule
{
  "content_ids": ["content-1", "content-2"],
  "platforms": ["tiktok", "youtube_shorts", "instagram_reels"],
  "max_posts_per_day": 3
}
```

### Scheduling Engine
The scheduler analyzes historical engagement data per platform to pick highest-engagement time slots. Posts are spread across days to avoid audience fatigue (configurable via max_posts_per_day). Confidence scores indicate how strongly the model believes in each recommendation.

### Engagement Metrics Tracked
- views, watch_time, likes, shares, comments, click_through, save_rate, follower_gain

## SERVICE 6: RED TEAM SECURITY SWARM (/v1/security)

### Purpose
Autonomous penetration testing agents that continuously attack all API endpoints to find vulnerabilities before external agents exploit them. Every finding is machine-readable so the Autonomous PM can auto-generate fix PRs.

### Attack Categories
- acl_bypass: Topic-level ACL evasion (the DJI Romo special)
- auth_probe: API key validation edge cases (missing, empty, malformed)
- injection: Payload injection (JSON, MQTT topics, path traversal)
- rate_limit_evasion: Sliding window limiter bypass attempts
- privilege_escalation: Cross-device/cross-agent unauthorized access
- schema_abuse: Malformed payloads, type coercion, oversized batches
- enumeration: Resource ID guessing, hidden endpoint probing
- replay: Request replay and token reuse
- dos_pattern: Expensive operation denial of service

### Endpoints
- POST /v1/security/scans → Launch full Red Team scan (async, returns 202)
- GET /v1/security/scans → List all historical scan reports
- GET /v1/security/scans/{scan_id} → Get full scan report with vulnerabilities
- GET /v1/security/scans/{scan_id}/vulnerabilities → Get vulnerabilities (filterable by severity)
- POST /v1/security/scans/quick → Quick scan for CI/CD gates (synchronous, critical vectors only)

### Example: Run security audit
```json
# 1. Launch full scan
POST /v1/security/scans
{
  "target_services": ["iot", "telemetry", "media", "comms", "factory"],
  "attack_categories": ["acl_bypass", "auth_probe", "privilege_escalation"],
  "intensity": "standard"
}

# 2. Check results
GET /v1/security/scans/{scan_id}
# Returns: score (0-100), severity breakdown, prioritized recommendations

# 3. Get actionable vulnerabilities
GET /v1/security/scans/{scan_id}/vulnerabilities?severity=critical
# Each vuln has: endpoint, evidence, CWE ID, remediation guidance

# 4. Quick scan for CI/CD
POST /v1/security/scans/quick
# Synchronous — use as deployment gate. Returns full report immediately.
```

### Security Score
0-100 scale. Penalties: critical=-25, high=-15, medium=-5, low=-2.
Score of 100 = fortress. Score of 0 = DJI Romo vacuum.

## ERROR FORMAT
All errors return JSON:
```json
{
  "detail": {
    "error": "error_code",
    "message": "Human/agent-readable description"
  }
}
```

## RATE LIMITS
- Default: 120 requests/minute per API key
- Batch endpoints count as 1 request regardless of payload size
- Rate limit headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset

## PAGINATION
All list endpoints support:
- `page` (int, default 1)
- `per_page` (int, default 50, max 200)

Response includes: `total`, `page`, `per_page`

## STATUS CODES
- 200: Success
- 201: Created
- 202: Accepted (async processing)
- 204: No content (successful delete)
- 400: Bad request
- 401: Missing API key
- 403: Invalid key or ACL violation
- 404: Resource not found
- 409: Conflict (duplicate resource)
- 429: Rate limited
- 500: Internal error
